# 🚀 项目报告：同城即时跑腿服务平台 (CityRunner)

### 一、 项目概述

本系统是一个基于 **Spring Boot + Vue 3** 架构的同城即时配送服务平台。系统实现了从用户下单、动态计费、骑手抢单到实时轨迹追踪的全链路业务闭环。项目核心难点在于**高并发下的抢单一致性**、**基于 LBS 的地理空间查询**以及**复杂的订单状态机管理**。

------

### 二、 技术栈架构

项目采用前后端分离架构，并进行多模块 Maven 聚合管理：

- **后端核心**: Java 17, Spring Boot 3.2, MyBatis-Plus
- **数据存储**: MySQL 8.0 (持久化), Redis 7.0 (缓存/分布式锁/LBS)
- **前端交互**: Vue 3, Pinia (状态管理), Element Plus (UI), AMap API (高德地图)
- **中间件/工具**: WebSocket (实时轨迹), Redisson (并发控制), Snowflake (全局唯一 ID)

------

### 三、 核心架构设计 (Maven Multi-Module)

为了实现代码的高内聚低耦合，项目分为以下模块：

- **`runner-common`**: 存放全局常量（位运算身份）、异常处理及工具类。
- **`runner-model`**: 统一管理 Entity、DTO、VO，保证数据模型一致性。
- **`runner-service`**: 封装核心业务逻辑（计费引擎、状态流转）。
- **`runner-api`**: 提供面向移动端和管理端的 RESTful 接口。

------

### 四、 核心业务实现细节

#### 1. 用户体系：位运算身份管理

采用**账号与角色分离**的设计。在 `u_user` 表中使用 `TINYINT` 类型的 `user_type` 字段，利用位运算实现多角色兼容：

- `01 (1)`: 普通消费者
- `10 (2)`: 骑手
- `11 (3)`: 既是消费者又是骑手

> **实现优势**：极大地节省了存储空间，并通过 `(user_type & 2) > 0` 的方式实现极速权限判定。

#### 2. 智能计费引擎 (Pricing Engine)

系统支持动态调价，运费根据**计费规则表**和**地图 API** 实时计算。计费公式如下：

$$TotalFee = BaseFee + \max(0, Dist - BaseDist) \times PricePerKm + WeightFee + NightMarkup$$

后端通过策略模式加载不同的 `FeeConfig`，确保运营人员可以随时调整起步价或夜间加价策略。

#### 3. 订单状态机与审计日志

系统严格控制订单的生命周期：`待支付 -> 待接单 -> 取货中 -> 配送中 -> 已送达`。

- **数据一致性**：使用状态机更新 SQL，防止状态回滚。
- **状态审计**：通过 `o_order_status_log` 记录每一次变更的操作人和时间戳，为 SLA（服务等级协议）分析提供数据支持。

#### 4. 高并发抢单方案 (Redis + Distributed Lock)

这是项目的技术高地：

- **附近搜单**：利用 Redis **GeoHash** 指令（`GEORADIUS`），实现骑手端秒级搜索方圆 5 公里内的待接订单。
- **并发控制**：使用 **Redisson 分布式锁** 锁定 `Order_ID`，确保在毫秒级并发抢单场景下，订单不会被重复领取，并保证数据库与缓存的最终一致性。

#### 5. 地理空间快照

在下单瞬间，系统将起点/终点的**文字地址及经纬度**冗余存储到订单表中。即使后续用户修改或删除了地址簿，历史订单的配送轨迹依然可追溯，确保了交易数据的静态快照特性。

------

### 五、 前端交互亮点

- **响应式地图**：集成高德地图 JS API，支持用户点击地图取点及自动逆地理编码。
- **实时追踪**：通过 **WebSocket** 实现骑手端位置实时推送到用户端，利用 Vue 3 的响应式特性平滑移动地图上的骑手图标。

------

### 六、 简历亮点归纳 (面试话术)

1. **架构思维**：采用 Maven 多模块设计，实现了业务逻辑与接口层的深度解耦。
2. **性能优化**：通过 Redis Geo 模块优化了地理位置查询性能，相比传统 SQL 查询效率提升数倍。
3. **可靠性**：设计了完善的支付流水及订单审计体系，能够有效应对掉单及业务纠纷。
4. **工程化**：使用 AOP 记录业务日志，并采用雪花算法解决分布式场景下的 ID 冲突。
